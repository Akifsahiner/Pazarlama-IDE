# Anti-Patterns

- **Don't:** Put "⭐ Star us on GitHub!" as the README H1 or Show HN title. **Why:** Reads as needy; HN and GitHub readers bounce before understanding value. **Cost:** 40–60% higher bounce; thread dies in <2h.
- **Don't:** Open 10 awesome-list PRs in one day with identical boilerplate. **Why:** Maintainers talk; labeled spam; all PRs closed. **Cost:** Burned list relationships for 12+ months.
- **Don't:** Cold-DM maintainers "Hey, love your project — can you star mine / add us to your README?" **Why:** Violates maintainer trust norms; screenshots get posted publicly. **Cost:** Reputation damage across OSS Twitter/IH.
- **Don't:** Launch Show HN before README install section works copy-paste. **Why:** HN clicks GitHub first; broken `npm install` = permanent roast in comments. **Cost:** One-shot traffic wasted; retry blocked 21d.
- **Don't:** Ship release notes as "Bug fixes and improvements" only. **Why:** Zero distribution signal; no blog/Dev.to hook; stars don't compound. **Cost:** Missed 200–800 long-tail sessions per release.
- **Don't:** Use 20+ GitHub topics including `awesome`, `hacktoberfest`, `javascript` on a niche CLI. **Why:** Topic spam reduces Explore relevance; looks amateur. **Cost:** Lower discoverability vs 5 precise topics.
- **Don't:** Hide license terms or use "source available" without SPDX. **Why:** Legal teams block eval; enterprise inbound stalls. **Cost:** Lost B2B pipeline from HN spike.
- **Don't:** Coordinate star swaps in Discord/Telegram groups. **Why:** GitHub ToS risk; obvious inflation; no real users. **Cost:** Account flags; credibility loss when stars don't convert to issues/downloads.
- **Don't:** Submit awesome-list PR without reading list contribution rules. **Why:** Instant close + "read CONTRIBUTING"; maintainer annoyance. **Cost:** Can't retry same list for 90d.
- **Don't:** Post star milestone at 47 stars with "Almost 50!" **Why:** Feels manufactured; community ignores. **Cost:** Wasted post; wait for organic threshold.
- **Don't:** Skip package registry publish and link only to GitHub raw install. **Why:** Registry search is how devs discover tools; `npm install X` must resolve. **Cost:** 50–70% of ICP never finds install path.
- **Don't:** File good-first-issue labels on issues that need core architecture knowledge. **Why:** First-time contributors fail; bad first impression. **Cost:** Zero contributions; negative CONTRIBUTING reputation.
- **Don't:** Run OSS launch and Product Hunt same 72h without repo freeze. **Why:** Split attention; README broken during PH traffic. **Cost:** Both channels underperform.
- **Don't:** Copy competitor README structure verbatim including their star badge widgets. **Why:** Looks derivative; unclear differentiation. **Cost:** Lower star-to-clone ratio.
- **Don't:** Ignore CHANGELOG after launch week. **Why:** Stars decay when repo looks abandoned; return visitors don't re-star. **Cost:** 15–25% star velocity drop by D+30 without releases.
