## Principles

1. **Evidence before claims.** Every feature, audience, and differentiator must cite a file path, route, or README quote. If you cannot cite it, label it `claims_needing_validation` — never assert it as fact.
2. **Read the repo like a buyer, not an engineer.** Prioritize `README`, landing routes (`app/page`, `pages/index`), pricing, and onboarding copy over internal modules. Buyers never see `src/lib/utils.ts`.
3. **ICP is a filter, not a vibe.** Name the role, company stage, trigger event, and budget authority. "Early adopters" and "tech-savvy users" are not ICPs — reject them.
4. **Positioning = category + outcome + wedge.** State what you replace, for whom, and the measurable outcome. One sentence a stranger can repeat after one read.
5. **Jobs-to-be-done beat feature lists.** Map 3–5 jobs the product helps complete, each tied to a route or workflow in the codebase.
6. **Competitors ground the wedge.** Name 2–4 real alternatives from `competitors` input or public knowledge; explain why this product wins on one axis only.
7. **Pricing signals willingness-to-pay.** Infer tier structure from `/pricing`, env vars, or README — flag if pricing is missing (blocks paid GTM).
8. **Routes reveal surface area.** Enumerate public routes; each cluster (marketing, app, docs, auth) informs channel fit and launch scope.
9. **Output is downstream fuel.** Product intelligence feeds landing copy, outreach, and launch assets — write for marketers who will not open the repo.
10. **Read-only discipline.** Inspect with Read/Grep/Glob only. Never mutate files in this skill.
